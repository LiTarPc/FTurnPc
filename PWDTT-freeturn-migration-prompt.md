# Промт для агента разработки: перевод PWDTT на ядро FreeTurn

> Этот файл — готовый промт-план. Его можно целиком скормить агентному инструменту
> (Claude Code, Cursor Agent, Aider и т.п.) в корне репозитория PWDTT.
> Ниже — не общие пожелания, а конкретные факты о коде обоих проектов
> (проверено чтением исходников), решение архитектурной развилки и пошаговый план.

---

## 0. Контекст (проверенные факты, не предположения)

### 0.1 Текущая архитектура PWDTT (luminescq/PWDTT)

- Десктоп-приложение на **Wails v2** (Go backend + TS/React-подобный frontend).
- `backend/orchestrator.go` — тонкий прокси между UI и **встроенным** Go-пакетом
  `client/core` (модуль `wg-turn-client/core`). Это **не внешний бинарник**, а
  собственная реализация протокола внутри самого PWDTT:
  - `client/core/creds_vkcalls.go`, `captcha_v2.go`, `captcha_v2_slider.go` — получение
    TURN-креды через VK Calls (включая обход капчи).
  - `client/core/obfs.go`, `wrap.go` — обфускация payload под RTP + ChaCha20-Poly1305
    (аналог `-obf-profile rtpopus` во FreeTurn).
  - `client/core/protocol.go`, `session.go`, `group.go`, `dispatcher.go` — сам туннель,
    воркер-группы, диспетчеризация пакетов.
  - `client/core/wgconfig.go` — **ключевая особенность**: ядро PWDTT умеет получать
    от сервера уже готовый WireGuard-конфиг через сам туннель (событие `wg_config`)
    и **автоматически применяет** его (`applyWGConfig` в backend, есть
    `linux_wg.go` / `windows_wg.go` / `macos_wg.go`). Пользователю не нужно вручную
    настраивать WireGuard — PWDTT сам поднимает интерфейс.
  - API ядра: `core.New(cfg) *Core`, `c.Start() (<-chan Event, error)`, `c.Stop()`,
    `c.SolveCaptcha(token)`, `c.AddTurnIPs(...)`/`GetTurnIPs()` (для исключения TURN-IP
    из туннеля, split-tunnel).
  - `core.Config`: `PeerAddr, Password, Hashes []string, Listen, TurnHost, TurnPort,
    DeviceID, Workers, CaptchaMode, MTU`.
  - `core.Event`: типы `state | log | event | stats | error`, событие `event/wg_config`
    несёт сырой WG-конфиг.
- `backend/deploy.go` — деплой **собственного сервера** на VPS по SSH: заливает
  встроенные (через `go:embed`, см. `backend/assets.go`) `deployScript` (bash) и
  `serverBinary` (`assets/server/wdtt-server`), запускает `deploy.sh` с env
  (`WDTT_ARGS`, `WDTT_DTLS_PORT`, `WDTT_WG_PORT`, `WDTT_SSH_PORT`). Сервер сам поднимает
  WireGuard-интерфейс на VPS и генерирует клиентский конфиг.
- Ссылки на серверы имеют собственный формат `wdtt://ip:dtlsPort:wgPort:proxyPort:
  password[:hash1,hash2,...][#name]` — это **свой протокол**, не совместимый с
  форматом FreeTurn.
- Лицензия PWDTT: **GPL-3.0**.

### 0.2 Архитектура ядра FreeTurn (samosvalishe/free-turn-proxy)

- Отдельный Go-проект, **не библиотека, а два бинарника**: `cmd/client`, `cmd/server`
  (собираются через `task build`). Вся реализация лежит в `internal/...` —
  **Go `internal`-пакеты недоступны для импорта из другого модуля** (Go запрещает
  это на уровне тулчейна). Значит "подключить как библиотеку" без форка/копирования
  кода **невозможно** — только (а) внешний процесс, либо (б) копирование исходников
  в дерево PWDTT (лицензия это разрешает, см. п. 0.3).
- FreeTurn — **не VPN и не "автопилот"**: это чистый транспортный слой.
  Клиент слушает `127.0.0.1:9000` (по умолчанию) и просто перекладывает UDP/TCP-пакеты
  в TURN/DTLS-туннель до сервера. WireGuard (или Xray/VLESS) — **отдельный,
  независимый процесс**, который пользователь (или скрипт) сам направляет на этот
  локальный порт. Автоматической доставки WG-конфига через сам туннель во FreeTurn
  **нет** — это принципиальное архитектурное отличие от PWDTT.
- CLI-флаги клиента (`docs/flags.md`), актуальные на момент проверки:

  | Флаг | По умолчанию | Назначение |
  |---|---|---|
  | `-listen` | `127.0.0.1:9000` | локальный адрес для WG/Xray |
  | `-peer` | обязателен | `host:port` сервера на VPS |
  | `-links` | ссылка `https://vk.com/call/join/...` |
  | `-n` | `10` | число параллельных TURN-потоков |
  | `-transport` | `tcp` | `tcp`\|`udp` до TURN-реле |
  | `-mode` | `udp` | `udp` (для WG) \| `tcp` (для Xray/sing-box) |
  | `-bond` | `false` | бондинг TCP-сессий (только `-mode tcp`) |
  | `-turn` / `-port` | из creds | форс IP/порт TURN |
  | `-obf-profile` | `none` | `none`\|`rtpopus` (RTP/opus + ChaCha20-Poly1305) |
  | `-obf-key` / `-gen-obf-key` | — | ключ обфускации, 32 байта hex |
  | `-manual-captcha` | `false` | ручной режим капчи |
  | `-streams-per-cred` | `10` | потоков на один кеш VK-креды |
  | `-dns-mode` | `auto` | `plain`\|`doh`\|`auto` |
  | `-dns-servers` | — | свои резолверы |
  | `-client-id` | авто | ID клиента (для allowlist на сервере) |
  | `-sub` | — | URL подписки со списком серверов |
  | `-debug` | `false` | подробные логи |

  Сервер: `-listen`, `-connect` (локальный backend, напр. `127.0.0.1:51820` для WG),
  `-mode`, `-obf-profile`, `-obf-key`, `-clients-file` (allowlist по client-id),
  `-debug`; плюс встроенные подкоманды `server clients add|remove|list`.
- Есть собственная система деплоя сервера по SSH (`docs/deploy.md`), собственный
  формат URI (`docs/uri.md`), подписки (`docs/sub.md`).
- Лицензия: **Happy Bunny License** — по сути MIT (разрешены use/copy/modify/merge/
  distribute/sublicense/sell) с двумя условиями: (1) сохранять copyright/уведомление
  во всех копиях, (2) "будь добр к кроликам" (шуточный, но формальный пункт).
  MIT-подобная лицензия совместима с включением в GPL-3.0 проект (PWDTT), в том числе
  при поставке бинарников FreeTurn внутри дистрибутива PWDTT — при условии сохранения
  текста лицензии/копирайта в поставке.

### 0.3 Итог сопоставления — где будет основная работа

FreeTurn закрывает функционально то же, что `client/core/{creds_vkcalls,captcha_v2,
obfs,wrap,protocol,session,group,dispatcher}.go` в PWDTT — то есть весь "движок"
получения TURN-кредов из VK Calls, туннелирования и обфускации. Но он **не закрывает**
UX-функцию PWDTT "получить готовый WG-конфиг от сервера и применить его в один клик" —
это логика, которую в PWDTT реализует связка `core/wgconfig.go` + `backend/*_wg.go`
+ протокол доставки конфига через сам туннель. Эту связку придётся переизобрести
на уровне оркестрации (см. Фазу 3), не трогая сам FreeTurn.

## 1. Архитектурное решение (обязательно закрыть перед началом работ)

Есть два пути. Агент должен явно выбрать один и зафиксировать выбор в
`docs/adr/0001-freeturn-integration.md` (создать такой ADR) перед тем как писать код.

### Вариант A — FreeTurn как внешний подпроцесс (рекомендуется)

- Кросс-компилировать `cmd/client` и `cmd/server` из free-turn-proxy (`task build:all`
  через goreleaser) под linux/amd64, windows/amd64, darwin/amd64+arm64.
- Бинарники кладутся в `assets/freeturn/<os>-<arch>/{client,server}(.exe)` и
  встраиваются в PWDTT через `go:embed`, как сейчас `wdtt-server` и `wintun.dll`.
- `backend/orchestrator.go` вместо `core.New(cfg)` запускает
  `os/exec.CommandContext(ctx, freeturnClientPath, args...)`, где `args` строятся
  из текущего `ConnectParams`/`ProfileData` по таблице маппинга (см. раздел 3).
- stdout/stderr процесса построчно парсятся и транслируются в существующие
  Wails-события `log`/`state_changed`/`error` (переиспользовать `wailsLogWriter`,
  адаптировав `classifyLevel` под формат логов FreeTurn — сверить реальный формат
  строк из `internal/logx`, при необходимости прочитать `internal/logx/*.go`).
- Готовность туннеля определяется не событием `wg_config` (которого не будет),
  а либо (a) фактом успешного локального `wg-quick`/handshake после применения
  заранее подготовленного конфига, либо (b) парсингом лог-строки FreeTurn о
  первом установленном TURN/DTLS-потоке (см. `activeConnectionCount >= 1` — такой
  индикатор явно упоминается в документации Android-клиента FreeTurn, аналог должен
  быть и в логах десктопного ядра — проверить в `internal/proxy/*`).
- Плюсы: минимальный риск, чистое разделение (обновления ядра FreeTurn = просто
  замена бинарника), не нужно тащить `internal/*` в свой модуль, лицензии не
  конфликтуют.
- Минусы: два процесса вместо одного, нужно аккуратно завершать дочерний процесс
  при Stop()/крэше, увеличивается размер дистрибутива (два набора платформенных
  бинарников).

### Вариант B — форк исходников FreeTurn внутрь модуля PWDTT

- Копировать `internal/*` из free-turn-proxy в `client/coreft/internal/*` (или
  публиковать форк как отдельный Go-модуль с публичными пакетами) и дергать
  Go-функции напрямую, без subprocess.
- Плюсы: один процесс, тоньше интеграция событий/логов, легче прокинуть
  собственный колбэк доставки WG-конфига прямо из кода FreeTurn.
- Минусы: обязанность синхронизировать форк при апстрим-обновлениях FreeTurn
  вручную (нет обновления "просто заменить бинарник"), больше кода в репозитории,
  выше риск разъехаться с апстримом по багфиксам протокола VK TURN (антидетект
  меняется чаще, чем хотелось бы — это активно поддерживаемый проект).

**Рекомендация агенту: начинать с Варианта A.** Он быстрее верифицируется, ниже риск
сломать текущий релиз, и даёт возможность так же просто откатиться на старое
`client/core`, если что-то в протоколе VK у FreeTurn отличается и ловит капчу/бан
иначе, чем текущее ядро PWDTT.

## 2. Пошаговый план работ

### Фаза 0 — Подготовка
1. Склонировать `github.com/samosvalishe/free-turn-proxy` как git submodule или
   отдельную build-зависимость (не как Go-модуль-импорт — см. п. 0.2 про `internal`).
2. Прочитать целиком `docs/quickstart.md`, `docs/modes.md`, `docs/flags.md`,
   `docs/deploy.md`, `docs/uri.md`, `docs/providers.md`, `docs/troubleshooting.md` —
   зафиксировать точный формат лог-строк клиента и сервера (нужно для парсера логов),
   а также точный формат данных, которые `-provider vk` ожидает в `-links`.
3. Собрать `task build` локально, убедиться что бинарники `client`/`server` рабочие,
   прогнать `-gen-obf-key`, посмотреть реальный вывод в stdout/stderr (форматы дат,
   уровней логов, маркеры готовности).
4. Написать ADR (см. раздел 1) с зафиксированным выбором Варианта A/B и обоснованием.

### Фаза 1 — Сборка и упаковка ядра FreeTurn
1. Добавить в CI (`.github/workflows/build.yml`, `build-macos.yml`, `release.yml`)
   шаг кросс-сборки `cmd/client`+`cmd/server` FreeTurn для всех целевых платформ,
   либо забирать готовые релизные бинарники из Releases free-turn-proxy — **выбрать
   один способ и не смешивать** (сборка из исходников надёжнее для воспроизводимости
   и лицензионной прозрачности).
2. Добавить `assets/freeturn/<platform>/client(.exe)`, `.../server` в дерево репо,
   `backend/assets.go` — добавить `go:embed` для нужной платформы (сейчас там уже
   есть паттерн для `wdtt-server`/`wintun.dll`, повторить его).
3. Добавить файл `assets/freeturn/LICENSE` (текст Happy Bunny License) и упомянуть
   FreeTurn/samosvalishe в `README.md`/`NOTICE` PWDTT — обязательное условие лицензии.

### Фаза 2 — Замена ядра в оркестраторе (backend/orchestrator.go)
1. Ввести интерфейс уровня оркестратора, например:
   ```go
   type TunnelEngine interface {
       Start(cfg EngineConfig) (<-chan Event, error)
       Stop()
       SolveCaptcha(token string)
   }
   ```
   Оставить `Event`/`EventType` как есть (уже используются по всему UI/трею) —
   это минимизирует изменения во frontend и в `linux_tray.go`/`windows_tray.go`.
2. Реализовать `FreeturnEngine` (новый файл `backend/engine_freeturn.go`),
   реализующий `TunnelEngine` через `os/exec`:
   - Построение `args []string` из `EngineConfig` по таблице маппинга (раздел 3).
   - Запуск с `CommandContext`, `Cancel: killProcess` (SIGTERM → таймаут → SIGKILL),
     отдельные горутины на чтение `Stdout`/`Stderr` построчно.
   - Парсер строк лога → `Event{Type: EventLog, ...}` + классификация уровня
     (адаптировать `classifyLevel`).
   - Детектор готовности туннеля (см. Фазу 3) → эмит `EventState{Status:"running"}`.
   - `Stop()` — контекст-cancel + ожидание завершения процесса с таймаутом.
   - `SolveCaptcha` — либо через stdin процесса (если FreeTurn поддерживает
     интерактивный ввод токена в `-manual-captcha` режиме — проверить в
     `internal/client/ish` и `internal/provider/vk`), либо через собственный
     локальный контрол-сокет, если FreeTurn такого не даёт "из коробки" — в этом
     случае может понадобиться доработка **самого** free-turn-proxy (PR в апстрим)
     для приёма капча-токена не через TTY, а через флаг/файл/сокет. Это отдельная
     задача, вынести в бэклог, не блокировать основную интеграцию (можно первой
     итерацией просто полагаться на `-provider vk` автоматический обход капчи,
     без ручного режима).
3. Оставить старый `client/core`-движок доступным под флагом (`EngineConfig.Engine:
   "legacy" | "freeturn"`), чтобы можно было A/B-сравнивать стабильность подключения
   и откатиться одной строкой конфига, если FreeTurn хуже проходит антибот VK в
   моменте (это меняется со временем на стороне VK).
4. Обновить `backend/app.go` — прокинуть выбор движка в `ConnectParams` (новое поле,
   с дефолтом `freeturn` после того как Фаза 5 подтвердит стабильность).

### Фаза 3 — Автоматическая доставка WireGuard-конфига (замена `wg_config`)
Так как FreeTurn не пушит WG-конфиг через туннель, нужно реализовать эквивалент
на уровне PWDTT, не трогая протокол FreeTurn:

1. **На этапе Deploy** (`backend/deploy.go`) сервер уже поднимает WireGuard и знает
   публичный ключ/адрес/порт. Расширить `deploy.sh` (или соответствующий скрипт
   FreeTurn из `docs/deploy.md`, если решите использовать штатный серверный
   деплой-флоу FreeTurn) так, чтобы он **возвращал** сгенерированный клиентский
   WG-конфиг (peer pubkey, endpoint, allowed-ips, client keypair) в stdout как
   маркированный блок (например между `---WG-CONFIG-BEGIN---`/`---END---`), который
   `sshExecStream`/`emit` в `deploy.go` уже умеет построчно читать.
2. Сохранять этот конфиг в `ProfileData` (новое поле `WGConfig string` в
   `ProfileData`/`profilePath`) в момент деплоя или при первом добавлении сервера
   по `wdtt://`-ссылке (нужно решить: либо ссылка `wdtt://` расширяется, либо вводится
   новый формат ссылки под FreeTurn/этот флоу — см. Фазу 4).
3. При `Start()`: сначала стартовать `FreeturnEngine` (поднимает локальный relay на
   `127.0.0.1:<listen-port>`), дождаться детектора готовности (см. п.2.2 выше — по
   логам FreeTurn о первом активном TURN/DTLS-потоке), **затем** применить сохранённый
   `WGConfig` через уже существующие `applyWGConfig`/`linux_wg.go`/`windows_wg.go`/
   `macos_wg.go` — они уже написаны и их менять не нужно, меняется только источник
   конфига (не событие ядра, а поле профиля).
4. Не забыть текущую фичу `AddTurnIPs`/`GetTurnIPs` (исключение TURN-серверов из
   туннеля, split-tunnel защита от петли маршрутизации) — у FreeTurn это тоже
   актуально (см. `builder.addDisallowedApplication`/`autoDetectInterfaceControl`
   в Android-клиенте, значит IP TURN-релея тоже нужно доставать и исключать).
   Источник IP на десктопе: либо распарсить его из аргумента `-peer`/`-turn` перед
   запуском (он у нас уже известен на стороне PWDTT, доставать из ядра не обязательно),
   либо перехватывать соответствующую лог-строку FreeTurn, если IP TURN-релея VK
   выбирается динамически самим FreeTurn (уточнить по `internal/provider/vk`).

### Фаза 4 — Формат ссылок и профили
1. Решить: либо расширить `wdtt://` (добавить поле версии/движка), либо ввести новый
   префикс, например `wdttft://`, чтобы явно отличать профили под FreeTurn-движок
   от легаси-профилей. Рекомендация: **версионировать существующий формат**
   (`wdtt://...#name` + необязательный query-подобный суффикс `?engine=freeturn`),
   чтобы не плодить сущности в UI (кнопка `+`, парсер вставки по Ctrl+V).
2. Обновить парсер ссылок (искать в `frontend/` — компонент, который матчит
   `wdtt://` при вставке) и Go-сторону парсинга, если она есть в backend.
3. Обновить `ProfileData` (структура в `orchestrator.go`) — добавить `Engine string`
   и `WGConfig string` (см. Фазу 3), сохранить обратную совместимость чтения старых
   `profiles/*.json` без этих полей (дефолт `Engine="legacy"`).

### Фаза 5 — UI/UX (минимальные изменения)
1. В настройках подключения добавить выбор ядра (Legacy / FreeTurn) — можно спрятать
   за "продвинутые настройки", т.к. по умолчанию агент должен предлагать FreeTurn
   только после того как Фаза 6 (тестирование) пройдена.
2. В Deploy-вкладке ничего в UX менять не обязательно, если Фаза 3.1 реализована как
   расширение существующего `deploy.sh` — пользователь не должен заметить разницы,
   кроме, возможно, новых полей вывода лога установки.
3. Проверить, что вкладка "VK Хеши" (1–4 хеша из `vk.com/call/join/<hash>`) остаётся
   актуальной — у FreeTurn это тот же формат `-links https://vk.com/call/join/...`,
   маппинг тривиален (см. таблицу в разделе 3).

### Фаза 6 — Тестирование и критерии приёмки
1. Юнит-тесты на построение `args` из `EngineConfig` (таблица кейсов: минимальный

## 3. Таблица маппинга параметров (для Фазы 2, построение args)

| Источник в PWDTT | Поле | Флаг FreeTurn client | Примечание |
|---|---|---|---|
| `ProfileData.PeerAddr` | | `-peer` | как есть |
| `ProfileData.Hashes` | | `-links` | FreeTurn принимает одну ссылку `-links`, а не массив хешей — либо брать первый хеш и формировать `https://vk.com/call/join/<hash>`, либо (если нужно 1–4 хеша, как в PWDTT) проверить, поддерживает ли `-links`/`-sub` несколько ссылок через запятую в актуальной версии — уточнить в `docs/uri.md`/`docs/sub.md`, при отсутствии поддержки — заводить upstream issue |
| `ConnectParams.Workers` | | `-n` | как есть |
| `ProfileData.Listen` / дефолт `127.0.0.1:9000` | | `-listen` | как есть |
| `ProfileData.TurnHost`/`TurnPort` | | `-turn` / `-port` | опционально, форс TURN-адреса |
| `ConnectParams.CaptchaMode` | | `-manual-captcha` | `true` только для режима `wv`/ручного; `auto`/`rjs` → флаг не передавать |
| (обфускация, была захардкожена как RTP/ChaCha20 в `client/core/obfs.go`) | | `-obf-profile rtpopus` `-obf-key <hex32>` | ключ обфускации — по аналогии с текущим PWDTT либо деривировать из `Password` (как сейчас `deriveWrapKey`), либо генерировать `-gen-obf-key` при первом деплое и хранить в профиле |
| нет прямого аналога сейчас | | `-mode udp` | обязательно `udp`, т.к. PWDTT поднимает WireGuard (не Xray) |
| нет прямого аналога сейчас | | `-transport tcp\|udp` | по умолчанию `tcp`, оставить дефолт, можно вынести в настройки позже |
| нет прямого аналога сейчас | | `-dns-mode auto` | оставить дефолт, не усложнять первую итерацию |
| нет прямого аналога сейчас | | `-client-id` | если включаете allowlist на сервере (`-clients-file`), генерировать и хранить стабильный ID в профиле; иначе не передавать |
| нет прямого аналога сейчас | | `-debug` | пробрасывать из уже существующего дебаг-тумблера приложения, если он есть, иначе всегда включать (логи и так фильтруются `classifyLevel`) |

Сервер (используется в обновлённом `deploy.sh`, Фаза 3):

| Параметр деплоя | Флаг FreeTurn server |
|---|---|
| DTLS-порт (`DtlsPort`) | `-listen 0.0.0.0:<port>` |
| WG-порт (`WgPort`) | `-connect 127.0.0.1:<wg-port>` |
| обфускация | `-obf-profile rtpopus -obf-key <тот же hex, что и у клиента>` |
| режим | `-mode udp` |
| allowlist (опционально) | `-clients-file /etc/freeturn/clients.json` + подкоманды `server clients add/remove/list` для управления |

## 4. Открытые вопросы, которые агент обязан явно закрыть до реализации (не додумывать)

1. Поддерживает ли текущий `-links`/`-sub` во free-turn-proxy несколько VK-хешей
   одновременно (в PWDTT их до 4 для отказоустойчивости)? Если нет — завести upstream
   issue/PR в `samosvalishe/free-turn-proxy` или временно ограничиться одним хешем
   в MVP-версии интеграции.
2. Как FreeTurn client принимает решённый капча-токен в интерактивном режиме —
   через stdin, файл, локальный HTTP/сокет? Нужно прочитать `internal/client/ish` и
   `internal/provider/vk` перед реализацией `SolveCaptcha` в `FreeturnEngine`.
3. Какой именно текст лог-строки сигнализирует "туннель установлен, первый поток
   активен" — нужно прочитать `internal/proxy/udprelay` и `internal/stats`, чтобы
   парсер готовности в `FreeturnEngine` не был основан на догадках.
4. Совпадает ли алгоритм обфускации `rtpopus` во FreeTurn с тем, что сейчас в
   `client/core/obfs.go`/`wrap.go` у PWDTT байт-в-байт — если нет, старые
   `wdtt://`-ссылки на существующие сервера пользователей **не будут** совместимы
   с новым движком до передеплоя сервера, это нужно явно предупредить в UI/README
   как breaking change при обновлении.
5. Зафиксировать конкретную версию/commit free-turn-proxy как git submodule pin
   (не "последняя" — проект активно меняется), обновлять осознанно с прогоном
   Фазы 6 заново.

> free-turn-proxy, либо задокументируй как явное ограничение MVP.
